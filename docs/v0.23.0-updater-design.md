# Nova v0.23.0 · 离线更新与版本管理 设计文档

> 目标: 让用户不再需要 PowerShell 手工覆盖。打开 Nova → 设置 → 更新 → 选择 `.nova-update` 文件 → 自动校验/安装/切换,失败可一键回退到任意历史稳定版。
> 兼容: 全量本地离线(无中心服务器假设),可接 Vault 迁移后的 auto-download,但 v0.23.0 先做手动导入更新包。

---

## 1. 目标与非目标

### 目标
1. **软件内一键更新**: 用户在 Nova 设置面板选 `.nova-update` 包 → 自动安装 → 重启生效。
2. **多版本并存**: 本地保留最近 N (默认 5) 个版本,不是覆盖式替换,而是版本化并存。
3. **一键回退**: 任何时刻可从"版本管理"选择某个历史版本作为当前运行版本。
4. **安装前完整性校验**: 每个 `.nova-update` 都带 SHA-256 清单 + 可选签名。校验失败不落地。
5. **失败自动回滚**: 安装过程崩溃、启动自检失败 → 自动切回上一个健康版本。
6. **完整离线工作**: 不依赖任何网络;所有更新包通过本地文件系统手动提供。

### 非目标 (留给 v0.23.x / v0.24)
- 自动从远端服务器拉取更新 (需要统一的 release server)
- 差量更新 / delta patch (v0.23.0 先做全量包)
- 用户数据库 schema 的前后迁移引擎 (目前 data 层是 Markdown + YAML,向前兼容)
- 多用户 / 多机器跨设备同步

## 2. 更新包格式 (.nova-update)

`.nova-update` 是一个标准 ZIP,扩展名改名以便系统关联。内部结构:

```
nova-v0.23.0.nova-update
├── manifest.json        # 必须,版本元信息 + SHA256 清单 + 最低兼容版本
├── signature.sig        # 可选 (v0.23.1+),对 manifest.json 做 ed25519 签名
├── payload/             # 实际要覆盖进 app_root 的文件树
│   ├── VERSION.txt
│   ├── backend/...
│   ├── electron/...     # 若改动了 Electron 主进程则包含
│   └── frontend_dist/...
└── migrations/          # 可选,安装前后要跑的脚本
    ├── pre-install.js   # 在新版本落地前执行 (如数据备份)
    └── post-install.js  # 在新版本落地后、Nova 重启前执行 (如配置迁移)
```

### `manifest.json` schema
```json
{
  "schema_version": 1,
  "package_id": "nova-v0.23.0-full",
  "target_version": "0.23.0",
  "min_base_version": "0.22.0",
  "release_channel": "stable",
  "released_at": "2026-05-04T00:00:00Z",
  "release_notes_md": "# v0.23.0\n- ...",
  "size_bytes": 2101234,
  "files": [
    { "path": "VERSION.txt",                "sha256": "...", "mode": 420 },
    { "path": "backend/services/vault_store.py", "sha256": "...", "mode": 420 },
    { "path": "frontend_dist/assets/index-xxx.js", "sha256": "...", "mode": 420 }
  ],
  "restart_required": true,
  "requires_electron_restart": false
}
```

- `min_base_version`: 拒绝从过老版本直接跨装。
- `restart_required`: 是否需要重启 Electron 主窗口(仅前端变动可热替)。
- `requires_electron_restart`: 是否必须重启 Electron 主进程(动了 main.js / preload.js 时 true)。

## 3. 版本化目录布局

当前固定路径 `app_root/` 改为"版本 slot + 软链接"模式:

```
app_root/
├── versions/
│   ├── 0.22.0/
│   │   ├── VERSION.txt
│   │   ├── backend/
│   │   ├── electron/
│   │   └── frontend_dist/
│   ├── 0.23.0/
│   │   └── ...
│   └── .index.json              # 所有已安装版本元信息
├── current -> versions/0.23.0/  # 软链接 / junction, Electron 和后端都从这里读
├── rollback_pointer.json        # 上一个健康版本,用于自动回滚
├── data/                        # 用户数据,不在版本化范围内
└── cache/updates/               # 下载/导入的 .nova-update 原件暂存
```

> Windows 用 `junction` 或 `mklink /D`,macOS/Linux 用 `symlink`。Electron `main.js` 通过 `path.resolve(__dirname, '..')` 解析的 `APP_ROOT` 会自动跟随 `current` 软链接。

**关键原则**: `data/` 永远在 `app_root/` 根部,不随版本切换移动。

## 4. 核心流程

### 4.1 安装新版本
```
[用户打开"设置 → 更新"]
      │
      ▼
[选择本地 .nova-update 文件]
      │
      ▼
[UpdaterService.importPackage(file)]
  1. 复制到 cache/updates/{pkg_id}.nova-update
  2. 打开 zip, 读 manifest.json
  3. 校验 schema_version / min_base_version
  4. 校验每个 file 的 sha256
  5. 展示版本说明给用户确认
      │
      ▼
[用户点击"安装"]
      │
      ▼
[UpdaterService.install(pkg_id)]
  1. rollback_pointer 指向当前版本 (安全网)
  2. 解压到 versions/{target_version}/ (原子 rename)
  3. 跑 pre-install.js (可选)
  4. 更新 versions/.index.json
  5. 原子切换 current -> versions/{target_version}
  6. 跑 post-install.js (可选)
  7. 若 restart_required => 弹窗提示用户点击"重启生效"
```

### 4.2 启动自检与自动回滚
```
Electron main.js 启动
      │
      ▼
[读 current 指向的版本, 载入 VERSION.txt]
      │
      ▼
[后端 FastAPI 自检 5s 内应答 /health]
      │ 成功                           │ 失败 / 5s 未响应
      ▼                                ▼
[versions/.index.json 标记 healthy]   [自动切 current -> rollback_pointer]
                                         │
                                         ▼
                                   [记录失败原因 -> crash.log]
                                   [弹窗提示用户"已回退到 X.Y.Z"]
```

### 4.3 手动回退
```
[用户打开"设置 → 版本管理"]
      │
      ▼
[列出 versions/.index.json 所有版本 + healthy 状态]
      │
      ▼
[用户选某个历史版本, 点"切换并重启"]
      │
      ▼
[UpdaterService.switchTo(version)]
  1. rollback_pointer 指向当前版本
  2. 原子切换 current 软链接
  3. 重启 Electron
```

## 5. 模块设计

### 5.1 后端 `backend/services/updater_service.py`
- `import_package(file_path) -> ImportResult`
- `install(package_id, *, pre_hooks=True, post_hooks=True) -> InstallResult`
- `list_versions() -> list[InstalledVersion]`
- `switch_to(version) -> SwitchResult`
- `get_rollback_target() -> str | None`
- `verify_package(file_path) -> VerifyResult`

### 5.2 Electron 侧 `electron/updaterBridge.js`
- 暴露 IPC:
  - `updater:open-file-picker`
  - `updater:verify`
  - `updater:install`
  - `updater:list-versions`
  - `updater:switch-to`
  - `updater:get-rollback-target`
  - `updater:restart`
- 负责跨平台 symlink / junction 原子切换
- 负责在 Nova 重启时读 `rollback_pointer` 做健康检查

### 5.3 前端 `nova-block/src/components/settings/UpdaterPanel.tsx`
- Tab 1 "检查更新": 上传 `.nova-update`,展示 manifest,确认安装
- Tab 2 "版本管理": 列表展示已安装版本 / 当前版本 / 最近回滚记录,支持切换
- Tab 3 "崩溃报告": 展示最近 crash.log 摘要(帮助用户判断是否要提 bug)

### 5.4 打包工具 `scripts/build-update-package.mjs`
- 输入: 一个 `v0.X.Y/` 目录 + `release-notes.md`
- 输出: 生成合规 `.nova-update` 文件,内嵌 manifest + SHA256 清单
- 可选: 接 ed25519 私钥签名

## 6. 兼容性与迁移

### 从 v0.22.0 (非版本化) 迁移到 v0.23.0 (版本化)
- 第一次运行 v0.23.0 时:
  1. 检测到 `app_root/` 里已经有裸 `backend/ electron/ frontend_dist/`,没有 `versions/`
  2. 创建 `versions/0.22.0/`,把现有文件搬进去
  3. 创建 `current` 软链接指向它
  4. 写 `versions/.index.json` 与 `VERSION.txt`
  5. 随后把 v0.23.0 本身作为一次 `install()` 落地,切换过去
- 这个 bootstrap 过程打包进 v0.23.0 的 `pre-install.js` 里

### 回退到 v0.22.x
- 因为 v0.22.x 不认识 `versions/` 结构,所以回退到 v0.22.x 需要"展平"操作:把 `current` 指向的内容复制回 `app_root/` 根部。
- 设计上 **v0.23.0 → 任意 v0.22.x 是单向的**(警告用户)。

## 7. 安全考量

- **校验链**: manifest.json 里每个文件 SHA-256 → 安装前逐个比对。
- **签名(v0.23.1+)**: 用 ed25519,公钥随 Nova 一起打包;私钥只在开发者本地。
- **权限**: 安装过程只写 `app_root/versions/` 和 `current`,不碰 `data/` 目录。
- **沙箱**: `pre-install.js` / `post-install.js` 在独立的 Node child_process 里跑,超时 60s 强制 kill。
- **崩溃自救**: 启动 2 次连续崩溃 → 自动切 rollback_pointer,并**禁用**该版本防止反复崩溃 loop。

## 8. 用户故事

### Story 1: 小王体验新版本
- 小王在群里拿到 `nova-v0.23.0.nova-update` 文件
- 打开 Nova → 设置 → 更新 → 选择文件 → 看到版本说明 → 点"安装" → 进度条走完 → 点"立即重启" → 新版本起来
- 整个过程零 PowerShell、零文件管理器

### Story 2: 更新后出问题
- v0.23.2 装完后 Nova 打开就白屏
- 启动自检失败 → 自动回退到 v0.23.1
- 用户下次打开就是稳定的 v0.23.1,顶部弹通知 "v0.23.2 安装后启动失败,已自动回退"

### Story 3: 主动回退
- 用户发现 v0.23.2 的某个新特性不好用,想回到 v0.23.0
- 打开"版本管理" → 看到列表 [0.23.2 ✓current, 0.23.1, 0.23.0, 0.22.0] → 点 0.23.0 → "切换并重启"
- Nova 重启,进入 0.23.0

## 10. 待确认问题 (开工前请拍板)

1. **版本保留数量**: 默认保留最近 5 个版本,其余自动清理。可接受?
2. **回退默认行为**: 启动失败自动回退 to last-healthy,还是只弹窗让用户决定?建议前者,因为多数用户不会主动操作。
3. **pre/post 钩子语言**: JavaScript(Node) vs Python。建议 Node,因为 Electron 自带,不依赖 Python 环境健康。
4. **是否要对 data/ 做自动备份**: 安装前 zip 一份 `data/` 到 `cache/backups/{timestamp}.zip`?会占空间但更安全。建议默认开启,可在设置里关闭。
5. **分发方式**: v0.23.0 只做"用户选本地文件",是否同期做"粘贴 URL 下载"?我倾向 **不做**,保持离线纯净,v0.24 再加远程 channel。
